# Lexiora 法律咨询助手 - 设计文档

> 基于多智能体（Multi-Agent）和GraphRAG的法律咨询助手
> 用途：毕业设计 + 研究生复试展示

---

## 1. 系统总体架构

系统分为四层，自下而上：

### 1.1 数据层
- **法律法规数据库**：结构化存储（法名/条款/内容）
- **案例数据库**：半结构化存储（案由/案情/判决结果）
- **知识图谱**：Neo4j存储实体和关系（法条↔案例↔法律概念）

### 1.2 GraphRAG层
- 从法规和案例中提取实体（罪名、法条、法律概念）和关系（引用、适用、相关）
- 向量索引 + 图索引双路检索
- 检索结果融合排序后提供给LLM

### 1.3 Agent层
- **协调Agent**：识别用户问题所属法律领域，分发到对应专业Agent
- **专业Agent**（民法/刑法/劳动法/行政法/其他）：各自专属prompt和检索策略
- **审查Agent**：对回答进行格式和准确性校验

### 1.4 交互层
- Streamlit Web界面，对话式问答
- 回答附带"依据"区域，展示引用的法条和案例
- 支持多轮对话（上下文保持）

### 1.5 数据流
```
用户提问 → 协调Agent意图识别 → 分发到专业Agent → 
GraphRAG检索相关知识 → 生成回答 → 审查Agent校验 → 返回用户
```

## 2. 组件详细设计

### 2.1 数据层组件

#### 2.1.1 数据采集模块
- 来源：中国法律法规数据库（国家法律法规数据库）和公开裁判文书数据集
- 第一阶段使用CAIL竞赛数据集，后期可扩展爬虫
- 数据格式：JSON/CSV

#### 2.1.2 数据处理模块
- **法规处理**：拆分为"法-章-条"三级结构
- **案例处理**：关键信息抽取（当事人、争议焦点、判决结果）
- 统一输出为结构化JSON格式

#### 2.1.3 知识图谱构建
- **三类节点**：法规（Law）、案例（Case）、法律概念（Concept）
- **五类关系**：引用（CITES）、适用（APPLIES）、相关（RELATED_TO）、属于（BELONGS_TO）、违反（VIOLATES）
- 通过LLM辅助抽取实体关系后写入Neo4j

### 2.2 GraphRAG模块

#### 2.2.1 实体抽取
- 从用户提问中提取法律相关实体（罪名、法条号、法律术语）
- 使用LLM进行命名实体识别（NER）

#### 2.2.2 双路检索
- **向量路**：用embedding模型将问题和文档编码，做相似度检索，覆盖语义相关但无关键词重叠的内容
- **图路径**：以提取的实体为起点，在Neo4j上做2-hop关系遍历，找出关联法条和案例

#### 2.2.3 结果融合
- 对两路结果去重、加权排序
- 取Top-K作为上下文提供给LLM

### 2.3 Agent层模块

#### 2.3.1 协调Agent（Coordinator）
- **输入**：用户自然语言问题
- **输出**：识别出的法律领域 + 关键意图标签
- **实现**：prompt-based分类，将问题归类到民法/刑法/劳动法/行政法/通用

#### 2.3.2 专业 Agent（Domain Agent）
- 每个领域一个Agent实例，共享基类架构但配置不同
- 内置系统prompt定义角色和回答格式
- 调用GraphRAG检索获取上下文
- 生成结构化的法律咨询回答（法条引用 + 分析 + 建议）

#### 2.3.3 审查 Agent（Reviewer）
- 检查回答是否包含法条引用
- 检查引用是否准确（与检索结果一致）
- 标记可能的幻觉内容

### 2.4 交互层

- Streamlit实现对话界面，保留对话历史
- 回答附带"依据"区域，展示引用的法条和案例
- 支持"追问"场景（多轮对话保持上下文）

## 3. 技术选型和开发工具链

| 层级 | 技术选型 | 备选方案 | 选择理由 |
|------|----------|----------|----------|
| LLM | 通义千问 (qwen-turbo / qwen-plus) | 智谱GLM-4 | 中文法律能力强，学生免费额度，API文档完善 |
| Embedding | text-embedding-v3 (通义) | BGE-M3 | 中英文双语，维度可选，与Qwen生态一致 |
| Agent框架 | LangGraph | 自主实现 | 专为多agent编排设计，状态图清晰可展示 |
| 图数据库 | Neo4j Community | NetworkX（内存方案） | 图RAG标准方案，社区版免费 |
| 向量存储 | FAISS | ChromaDB | 轻量、本地、无额外服务依赖 |
| Web框架 | Streamlit | FastAPI + 前端 | 零前端知识要求，快速搭建，适合答辩演示 |
| 项目管理 | uv | conda/venv | 现代Python包管理，快速可靠 |
| 开发语言 | Python 3.11+ | — | 上述所有库的原生语言 |

## 5. 开发阶段规划

### 阶段一：GraphRAG核心（约3周）
- 搭建法律知识图谱（法规、案例的实体和关系）
- 实现基础的检索增强生成
- **里程碑**：可以回答"XXX法第几条是什么"类问题

### 阶段二：单Agent + 工具调用（约3周）
- 一个"协调员"agent，能调用检索工具、分析工具
- 引入意图识别和回答模板
- **里程碑**：可以理解用户意图并给出结构化法律建议

### 阶段三：多Agent角色分工（约4周）
- 扩展为民法/刑法/劳动法等专业agent
- 协调agent路由分发
- **里程碑**：完整的法律咨询对话系统

## 6. 架构设计原则

### 6.1 业务逻辑与UI严格分离
- Streamlit层只做展示，不混入业务逻辑
- 所有核心逻辑封装在独立的Python模块中
- 便于未来迁移到FastAPI时只需替换app层

### 6.2 模块化设计
- 每个模块有明确的职责和接口
- 可以独立测试和理解
- 文件保持聚焦，避免过大文件

### 6.3 可解释性
- 回答附带法条引用和案例依据
- 检索过程可追溯
- 适合答辩展示和学术评审
